## Issues faced and fixes applied (in chronological order)

### 1. Default OpenHarness system prompt was poisoning the session

**Symptom:** With `--system-prompt "$(cat ...)"`, Gemma4 was still acting like
a generic coding assistant — drilling down per-device after `0 results`,
ignoring SOC-specific instructions.

**Root cause:** `build_runtime_system_prompt` in
`src/openharness/prompts/context.py` was injecting MORE sections AFTER the
custom prompt: skills catalogue (`commit`, `debug`, `plan`, `review`, etc.),
memory, reasoning settings, CLAUDE.md. The assembled prompt was 76 lines (4413
chars) instead of the expected 27 lines (3000 chars). Gemma4 was reading
instructions from THREE conflicting sources.

**Fix:** Modified `context.py` so that when `settings.system_prompt` is set
(via `--system-prompt`), it returns ONLY the user's prompt + environment
section. All other auto-injected sections are skipped.

```python
def build_runtime_system_prompt(...):
    sections = [build_system_prompt(custom_prompt=settings.system_prompt, cwd=str(cwd))]
    # If a custom system prompt was provided, return it stripped of injected
    # context. The custom prompt is the user's explicit replacement.
    if settings.system_prompt:
        return sections[0]
    # ... rest of original logic ...
```

**Verified by:** Re-printing the assembled prompt — dropped from 4413 chars
back to ~3300 chars (just custom prompt + small environment section).

### 2. Gemma4 forgetting `closure_notes` in `fortisoar_resolve_alert`

**Symptom:** After investigation, Gemma4 called
`fortisoar_resolve_alert(alert_id="Alert-XXXX")` with no closure_notes.
Pydantic `min_length=20` rejected with "field required". Model couldn't
recover and stalled.

**Root cause:** Smaller models can't reliably synthesize multi-paragraph
closure notes from scratch when faced with a "field required" error. The
model's recovery strategy is poor.

**Fix:** Made `closure_notes` optional at the schema level (`default=""`).
When notes are missing or shorter than 20 characters, the tool now:
1. Fetches the alert (already needed for tenant/status checks)
2. Builds a complete closure-notes template from structured fields
   (alert_id, name, src_ip, dst_ip:port, device, ADOM, severity, action)
3. Returns the template as an error wrapped in `--- COPY THIS TEMPLATE ---`
   markers, with a literal next-call template

The model's job becomes "copy the template into the next call" — which even
a 4B-active model handles reliably.

**Code:** `_build_closure_notes_template()` in `fortisoar_resolve_alert_tool.py`.

**Note:** In practice, after the compact alert format fix below, Gemma4
reliably writes its own notes on the first call. The template fallback never
actually triggered in the successful runs.

### 3. The verbose alert format was the silent killer

**Symptom:** Gemma4 would call `fortisoar_get_alert`, get the response, and
then produce a completely empty assistant turn — `{"role": "assistant",
"content": []}`. Investigation never started.

**Root cause:** The original `fortisoar_get_alert` formatter produced ~174
lines / ~6KB of output for one alert. Most of it was empty fields
(`mitre_attack_id: -`, `user_name: -`, `file_hash: -`, `command_line: -`,
etc.) and a 3KB raw `sourcedata` JSON dump. The smaller model couldn't reason
over that much noise on the second turn — it just gave up.

**Fix:** Rewrote `_format_alert` in `fortisoar_get_alert_tool.py` to be
aggressively compact:
- One-line header with `status / severity / type / tenant / source_tool`
- Correlation hints **first** (the most actionable section)
- All empty fields hidden entirely
- `description` capped at 600 chars
- Raw `sourcedata` JSON dump REMOVED from the formatter (still in the alert
  dict if someone wants it; just not in the formatted output)

**Result:** Output dropped from **174 lines / 6KB** to **32 lines / 1.8KB** —
70% reduction. After this change, Gemma4 reliably completes the workflow
end-to-end.

**This was THE fix that made the difference.** Everything else helped, but
this is what unlocked second-turn reasoning.

### 4. Tool descriptions encouraged per-device drilling

**Symptom:** When `faz_query_logs` returned 0 results from `All_FortiGate`,
Gemma4 would call `faz_get_devices` and start querying each FortiGate one by
one (94 devices × 4 queries = 376 wasted API calls, would never converge).

**Root cause:** The `device` parameter description said "Use a specific
device name from faz_get_devices to scope the query." That nudged the model
to drill down. And `faz_get_devices` had no warning that it shouldn't be used
during alert investigation.

**Fix:** Rewrote three field/tool descriptions:
- `faz_query_logs.device`: "Leave as the default 'All_FortiGate' in almost
  every case... Do NOT iterate over individual devices: it's slower,
  wasteful, and produces the same answer."
- `faz_query_security_events.device`: same wording
- `faz_get_devices` top-level description: "USE THIS ONLY WHEN: the user
  explicitly asks 'what devices are in this ADOM' or you need to map an IP to
  a specific device by hand. DO NOT use this during alert investigation..."

### 5. The temperature=0 mistake (CAUTIONARY TALE)

**Symptom:** I read session snapshots showing the same alert (Alert-107589)
"failing" 5 times in a row then "succeeding" on the 6th try. I concluded:
"non-deterministic — must be temperature 0.7."

**My wrong fix:** Added `"temperature": 0.0` to the OpenAI client params in
`openai_client.py`.

**Why it was wrong:** I never reset the alert state between trials. Trial 1
successfully closed the alert. Trials 2-5 then saw `status=Closed` and
correctly skipped the resolve step (the resolve tool's "already Closed" guard
returns an error and the model produces empty text — which is THE CORRECT
BEHAVIOR). I mistook correct behavior for failure.

**The user caught it:** "I think your A/B test are not correct. did you make
sure the alert is in open status before you run the non-interactive Gemma4
closure testing?" — and they were right.

**Proper test (with state reset between trials):** Built a test script that
PUTs the alert back to Open via direct API before each `oh -p` invocation.
Result on Alert-107558 with default temperature: **5/5 successes**. Not 1/5,
not 4/5 — five out of five.

**Conclusion:** Temperature was NOT the cause. Reverted the change. The
actual cause was the verbose alert format (issue #3 above), which I'd already
fixed.

**Lesson saved to memory:** `feedback_debug_order.md` — exhaust software-level
fixes before touching model parameters. Always reset state between A/B
trials. Always verify against the live system, not session snapshots alone.

## Open questions / what's NOT done

1. **Interactive mode controlled test.** Never ran a proper reset-between-trials
   A/B in interactive mode. The user's complaints about interactive mode were
   probably from BEFORE the compact alert format fix took effect, but we
   haven't conclusively proven interactive mode works equally well.

2. **"Should NOT close" alert testing.** We've only tested alerts that the
   agent SHOULD resolve. We have not verified what Gemma4 does when:
   - The action was `accept`/`allow`/`pass` (the traffic actually went through)
   - There are 100+ events from the same source (sustained attack)
   - The destination shows lateral movement after the event
   - The alert is recent (today) with real correlation data
   This is the highest-stakes failure mode. The prompt has rules for it but
   smaller models sometimes ignore guardrails when they pattern-match.

3. **Alert age sanity check.** Alerts older than ~7 days have aged out of FAZ
   retention. The "0 results across all queries" looks like clean evidence to
   a small model, but it's actually "no data exists." Suggested fix: surface
   `alert_age_days` and `correlation_recommended` in the structured hints.

4. **`fortisoar_escalate_alert` tool.** Not built. The user said they have
   mature escalation procedures (Teams/email webhooks) that fire on status
   changes from Open → Investigating/Pending. When ready, build a simple
   tool that just changes status (no email logic needed).

5. **Phase 2 cron-driven autonomous mode.** Wrap everything in an agent
   definition + cron schedule. Straightforward extension once Phase 1 is
   solid.
